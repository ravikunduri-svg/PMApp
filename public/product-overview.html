Scoring System

The 6 PM Dimensions

Every question, every answer, and every score maps to one or more of these six dimensions. Each is rated 1–10 — null means the question doesn't test that dimension.

🔭
Problem Framing
Scoping and reframing the problem before jumping to solutions. Can you identify what's actually being asked?
🧠
User Empathy
Behavioral (not demographic) user understanding. What does the user actually feel and do — not just who they are?
🏗️
Structured Thinking
Logical organization of ideas. MECE frameworks, clear 2×2s, explicit reasoning chains.
⚖️
Prioritization
Hard choices with explicit tradeoffs named. Not everything is P0. What gets cut and why?
📈
Metrics Reasoning
Defining meaningful success metrics. North star vs. guardrail vs. diagnostic — and knowing the difference.
🗣️
Communication Clarity
Clear, concise, compelling communication. Can you explain a complex product decision in 30 seconds?
Scoring note: Each question targets 2–4 dimensions. Only tested dimensions receive a score and feedback. null = not tested by this question — it won't drag down your average.

Unique Feature

JD Readiness Check Flow

The only PM prep tool that tells you if you're ready for a specific job. Paste the JD — get a readiness percentage, a dual radar chart, and a gap-by-dimension action plan.

1

Gate check — minimum 3 completed tests

JD analysis is only unlocked after 3 completed practice sessions. Low test count = low confidence in your scores = unreliable gap analysis. Confidence indicator: Low (<3) · Moderate (3–5) · High (6–10) · Very High (>10).

2

Parse the JD via Groq

The raw job description text (≥100 chars) is sent to llama-3.3-70b-versatile with a structured prompt. The LLM extracts: parsed_role name, 6 dimension importance scores (1–10), and inferred archetype (consumer / b2b / technical).

3

computeReadiness() — pure math, no LLM

Gap per dimension = max(0, required − user_score). Total gap summed across all 6 dims. Readiness % = 1 − (total_gap / max_possible_gap) × 100. Verdict: ready (≥80%) · almost (60–79%) · not yet (<60%).

4

Dual-overlay radar chart

Recharts RadarChart renders two overlapping polygons: indigo fill for your scores, amber dashed line for role requirements. Visual gaps are immediately obvious. Each dimension shows the exact score delta.

5

Gap bars → targeted practice

Dimensions sorted by gap size (largest first). Each bar links directly to the relevant practice questions — so users go from "I'm 3 points behind on Metrics Reasoning" to "here's the next question to fix that" in one click.

Cost target: ~₹1 per JD check (one Groq call, ~500 token prompt). computeReadiness() is pure math — zero LLM cost for the readiness calculation itself.
Database Schema

Core Data Model

Three application tables plus Supabase's built-in auth. All tables have Row Level Security enabled — users can only see their own data.

Table Key columns Purpose Written by
profiles user_id, archetype, initial_dimension_scores (JSON), created_at Stores archetype selection and seed dimension scores from onboarding quiz /onboarding page · auto-created by trigger on signup
practice_sessions id, user_id, archetype, status, questions (JSON array), question_results (JSON array), scores (6-dim JSON), overall_score One row per test attempt. question_results accumulates as user answers each question. /api/select-questions creates · /api/score-answer appends
readiness_checks id, user_id, jd_text, parsed_role, role_requirements (JSON), gap_analysis (JSON), recommendation, readiness_pct One row per JD analysis. Stores full gap breakdown for trend tracking. /api/parse-jd
pm_documents id, content, embedding (vector 384), doc_type, source_id, metadata RAG knowledge base. Questions and module sections embedded as 384-dim cosine vectors. rag/ingest.py (offline pipeline)
pm_ingest_log source_id, file_path, content_hash, chunk_count, ingested_at Tracks which files have been embedded. SHA-256 hash enables incremental re-ingest. rag/ingest.py
